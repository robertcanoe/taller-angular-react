¿Qué es Angular en 2026?

Imagina que quieres construir Gmail o Spotify desde cero. Angular es el kit completo: no sólo ladrillos, sino también el plano, las herramientas y el manual.

  • Framework completo — no necesitas buscar librerías para cada cosa
  • TypeScript — JavaScript con tipos: detecta errores antes de ejecutar
  • Componentes standalone — piezas de la app autosuficientes y reutilizables
  • Signals — reactividad inteligente: la pantalla se actualiza sola cuando cambian los datos

Angular: La Evolución

Si has oído que Angular es "muy complicado", eso era antes. La versión moderna es radicalmente más simple.

❌ Angular Antiguo (v2-v14)

NgModules (“contenedores burocráticos” obligatorios), Zone.js (libreria extra), mucho código repetitivo para cosas simples

✅ Angular 20/21 (hoy)

Standalone (componentes autosuficientes), Signals (reactividad simple), menos código, mucho más rendimiento

Componentes en Angular

La unidad mínima de tu aplicación

Un componente = HTML + estilos CSS + lógica TypeScript empaquetados juntos.
Como una pieza de LEGO: autosuficiente, reutilizable y combinable con otras piezas.

Antes vs Ahora: cómo se declara un componente

Antes había que registrar cada componente en un "módulo" intermediario (NgModule). Ahora el componente se gestiona solo.

❌ Angular Antiguo

@NgModule({

declarations: [UserComponent],

imports: [CommonModule],

})

export class UserModule {}

✅ Angular 20/21

@Component({

standalone: true,

imports: [CommonModule],

template: `<h2>User!</h2>`

})

export class UserComponent {}

Tu Primer Componente Standalone

💡 selector: el tag HTML que usarás en otros templates  |  standalone: true: se gestiona solo  |  template: el HTML que renderiza

import { Component } from '@angular/core';

@Component({

selector: 'app-saludo',

standalone: true,

template: `<h1>¡Hola {{ nombre }}!</h1>`

})

export class SaludoComponent {

nombre = 'Angular 21';

}

Interpolación: mostrar datos en pantalla

💡 Las dobles llaves {{ }} son "huecos" en el HTML que Angular rellena con el valor actual de una variable del componente.

export class ProductoComponent {

precio = 1999;

// Template

<p>Precio: {{ precio }}€</p>

Property Binding: pasar datos al HTML

💡 Los corchetes [propiedad] le dicen a Angular: “evalua este valor como JavaScript y pásale al atributo del elemento”. Es un enlace de datos en una dirección: de la clase al HTML.

export class ImagenComponent {

imagenUrl = 'assets/logo.png';

descripcion = 'Logo Angular';

deshabilitado = false;

}

// Template

<img [src]="imagenUrl" [alt]="descripcion">

<button [disabled]="deshabilitado">Click</button>

Event Binding: reaccionar a acciones del usuario

💡 Los paréntesis (evento) capturan eventos del usuario (clic, tecla pulsada, ratón...) y ejecutan un método. Es un enlace en la otra dirección: del HTML a la clase.

export class ContadorComponent {

contador = 0;

incrementar() {

this.contador++;

}

}

// Template

<p>Contador: {{ contador }}</p>

<button (click)="incrementar()">+1</button>

Two-Way Binding: sincronización automática

💡 [(ngModel)] es la combinación de [ ] y ( ): cuando el usuario escribe en el input, la variable se actualiza; y cuando la variable cambia en código, el input también se actualiza. El dato fluye en ambas direcciones.

import { FormsModule } from '@angular/forms';

@Component({

standalone: true,

imports: [FormsModule],

template: `<input [(ngModel)]="nombre">`

})

export class FormComponent {

nombre = '';

}

Angular Signals

Reactividad moderna: la pantalla se actualiza sola

Como una hoja de cálculo de Excel: cuando cambias una celda, todas las fórmulas que dependen de ella se recalculan automáticamente.

¿Qué son los Signals?

Una Signal es una variable “inteligente”: Angular sabe exactamente dónde se usa y, cuando su valor cambia, actualiza sólo esas partes de la pantalla — sin recargar nada más.

  • Sistema de reactividad nativo de Angular (sin librerías externas)
  • Notifican automáticamente cuando cambian
  • Mejor rendimiento: sólo actualiza lo necesario
  • Más fácil de entender que los Observables de RxJS

Crear un Signal

💡 A diferencia de una variable normal, lees una signal llamando a la función: contador() en lugar de sólo contador. Esa llamada registra la dependencia.

import { Component, signal } from '@angular/core';

export class ContadorComponent {

// Signal writable

contador = signal(0);

incrementar() {

this.contador.set(this.contador() + 1);

}

}

Leer y Actualizar Signals

// Leer valor (se ejecuta como función)

const valor = this.contador();

// Actualizar valor

this.contador.set(10);

// Actualizar basado en valor actual

this.contador.update(val => val + 1);

// En template

<p>Contador: {{ contador() }}</p>

Computed Signals

import { computed } from '@angular/core';

export class CarritoComponent {

productos = signal([

{ nombre: 'Laptop', precio: 999 },

{ nombre: 'Mouse', precio: 29 }

]);

total = computed(() =>

this.productos().reduce((sum, p) =>

sum + p.precio, 0)

);

}

Effect: Reaccionar a Cambios

import { effect } from '@angular/core';

export class LogComponent {

contador = signal(0);

constructor() {

effect(() => {

console.log('Contador cambió:',

this.contador());

});

}

}

@Component({

Control Flow Moderno

@if, @for, @switch

Controla qué partes del HTML se muestran u ocultan, y cómo se repiten los elementos — directamente dentro del template, con una sintaxis limpia similar a JavaScript.

@if — Mostrar u ocultar según condición

💡 Reemplaza al antiguo *ngIf. La nueva sintaxis con @if es más natural (igual que un if de JavaScript) y no requiere importar ningún módulo.

// Antes: *ngIf

<div *ngIf="usuario">Hola {{ usuario }}</div>

// Ahora: @if

@if (usuario) {

<p>Hola {{ usuario }}</p>

} @else {

<p>No hay usuario</p>

}

@for — Repetir elementos para una lista

💡 Reemplaza el antiguo *ngFor. El track es obligatorio: indica a Angular cómo identificar cada elemento para actualizar sólo los que cambian (optimización de rendimiento).

// Antes: *ngFor

<li *ngFor="let item of items">{{ item }}</li>

// Ahora: @for con track

@for (item of items; track item.id) {

<li>{{ item.nombre }}</li>

} @empty {

<p>No hay items</p>

}

@switch - Múltiples Casos

@switch (estado) {

@case ('cargando') {

<p>Cargando...</p>

}

@case ('exito') {

<p>¡Datos cargados!</p>

}

@case ('error') {

<p>Error al cargar</p>

}

@default {

<p>Estado desconocido</p>

}

}

Pipes: transformar datos al mostrarlos

💡 Un pipe transforma un valor solo en la vista, sin modificar el dato original. Se usan con el símbolo | dentro de la interpolación.

<!-- Mayúsculas -->

<p>{{ nombre | uppercase }}</p>

<!-- Fechas -->

<p>{{ hoy | date:'dd/MM/yyyy' }}</p>

<!-- Moneda -->

<p>{{ precio | currency:'EUR' }}</p>

<!-- JSON -->

<pre>{{ objeto | json }}</pre>

Routing en Angular

Navegar entre páginas sin recargar

Cada URL de tu app muestra un componente diferente.
Como las páginas de un libro, pero sin que el navegador tenga que hacer una nueva petición al servidor.

Configurar Rutas

💡 Las rutas son como una "tabla de contenidos": dictan qué componente se renderiza según la URL visible en el navegador. ** es el comodín para URLs que no existen (página 404).

// app.routes.ts

import { Routes } from '@angular/router';

export const routes: Routes = [

{ path: '', component: HomeComponent },

{ path: 'productos', component: ProductosComponent },

{ path: 'producto/:id', component: DetalleComponent },

{ path: '**', component: NotFoundComponent }

];

// main.ts

bootstrapApplication(AppComponent, {

providers: [provideRouter(routes)]

});

Router Outlet: el "hueco" donde aparecen las páginas

💡 <router-outlet /> es el marcador de posición en tu template donde Angular insertará el componente correspondiente a la URL actual. El nav y el menú quedan fijos; sólo cambia el contenido del outlet.

// app.component.ts

import { RouterOutlet } from '@angular/router';

@Component({

standalone: true,

imports: [RouterOutlet],

template: `

<nav><!-- Menú --></nav>

<router-outlet />

`

})

export class AppComponent {}

RouterLink: navegar sin recargar la página

💡 Usa routerLink en lugar de href. La diferencia: href recarga toda la página; routerLink hace una navegación interna instantánea (SPA — Single Page Application).

import { RouterLink } from '@angular/router';

@Component({

imports: [RouterLink],

template: `

<a routerLink="/">Home</a>

<a routerLink="/productos">Productos</a>

<a [routerLink]="['/producto', id]">

Ver Detalle

</a>

`

})

Capturar Parámetros

import { ActivatedRoute } from '@angular/router';

export class DetalleComponent {

route = inject(ActivatedRoute);

id = signal('');

ngOnInit() {

this.id.set(

this.route.snapshot.params['id']

);

}

}

Lazy Loading: cargar bajo demanda

💡 En lugar de enviar todo el código de la app de golpe, con Lazy Loading sólo se descarga cada sección cuando el usuario navega a ella. Resultado: carga inicial más rápida. Usando loadComponent Angular hace el split automáticamente.

export const routes: Routes = [

{

path: 'admin',

loadComponent: () =>

import('./admin/admin.component')

.then(m => m.AdminComponent)

},

{

path: 'dashboard',

loadChildren: () =>

import('./dashboard/routes')

}

];

HTTP Client

Tu app hablando con el mundo exterior

Una API es como un camarero: tú le pides datos (GET), le envías datos nuevos (POST), o le dices que borre algo (DELETE). Angular incluye un servicio HTTP listo para usar.

Configurar HTTP Client

// main.ts

import { provideHttpClient } from '@angular/common/http';

bootstrapApplication(AppComponent, {

providers: [

provideRouter(routes),

provideHttpClient()

]

});

Crear un Servicio HTTP

💡 Un servicio es una clase reutilizable que no es un componente. Su trabajo es encapsular la lógica de negocio (como las llamadas HTTP) para que los componentes no se ensucien con ese código.

import { Injectable, inject } from '@angular/core';

import { HttpClient } from '@angular/common/http';

@Injectable({ providedIn: 'root' })

export class ProductosService {

http = inject(HttpClient);

url = 'https://api.example.com/productos';

obtenerTodos() {

return this.http.get<Producto[]>(this.url);

}

obtenerPorId(id: number) {

return this.http.get<Producto>(`${this.url}/${id}`);

}

}

Usar el Servicio en un Componente

💡 inject() obtiene una instancia del servicio (Angular la crea y gestiona automáticamente). .subscribe() espera a que llegue la respuesta HTTP y ejecuta el callback con los datos.

export class ProductosComponent {

service = inject(ProductosService);

productos = signal<Producto[]>([]);

ngOnInit() {

this.service.obtenerTodos()

.subscribe(datos => {

this.productos.set(datos);

});

}

}

// Template

@for (p of productos(); track p.id) {

<div>{{ p.nombre }}</div>

}

Signals + HTTP: el patrón más moderno

💡 toSignal() convierte un Observable (lo que devuelve HTTP) en una Signal. Así puedes usar los datos directamente en el template con productos(), sin necesidad de suscribirte manualmente.

import { toSignal } from '@angular/core/rxjs-interop';

export class ProductosComponent {

service = inject(ProductosService);

// Convertir Observable a Signal

productos = toSignal(

this.service.obtenerTodos(),

{ initialValue: [] }

);

// Uso directo en template

// @for (p of productos(); track p.id)

}

App de Tareas con API Real

Ponemos en práctica todo lo aprendido: componentes, signals, routing y HTTP, integrados en una app funcional.

Estructura del Proyecto

  • Listar tareas desde JSONPlaceholder API
  • Agregar nuevas tareas
  • Marcar como completadas
  • Eliminar tareas

Definir el Modelo

// models/tarea.interface.ts

export interface Tarea {

id: number;

title: string;

completed: boolean;

userId: number;

}

Servicio de Tareas

@Injectable({ providedIn: 'root' })

export class TareasService {

http = inject(HttpClient);

url = 'https://jsonplaceholder.typicode.com/todos';

obtenerTareas() {

return this.http.get<Tarea[]>(this.url);

}

crear(tarea: Partial<Tarea>) {

return this.http.post<Tarea>(this.url, tarea);

}

actualizar(id: number, tarea: Partial<Tarea>) {

return this.http.put<Tarea>(`${this.url}/${id}`, tarea);

}

}

Componente de Tareas (1/2)

export class TareasComponent implements OnInit {

service = inject(TareasService);

tareas = signal<Tarea[]>([]);

nuevaTarea = ''; // string normal (compatible con ngModel)

pendientes = computed(() =>

this.tareas().filter(t => !t.completed).length

);

ngOnInit() {

this.service.obtenerTareas()

.subscribe(datos => {

this.tareas.set(datos.slice(0, 10));

});

}

}

Componente de Tareas (2/2)

agregarTarea() {

const nueva = {

title: this.nuevaTarea,

completed: false,

userId: 1

};

this.service.crear(nueva).subscribe(t => {

this.tareas.update(lista => [...lista, t]);

this.nuevaTarea = '';

});

}

toggleCompletada(tarea: Tarea) {

const actualizada = { ...tarea,

completed: !tarea.completed };

this.tareas.update(lista =>

lista.map(t => t.id === tarea.id ? actualizada : t)

);

}

Template de Tareas

💡 [(ngModel)] sincroniza el input con la variable. [class.completada] aplica una clase CSS condicionalmente. pendientes() es una computed signal.

<div class="container">

<h1>Mis Tareas</h1>

<p>Pendientes: {{ pendientes() }}</p>

<input [(ngModel)]="nuevaTarea"

placeholder="Nueva tarea">

<button (click)="agregarTarea()">Agregar</button>

<ul>

@for (tarea of tareas(); track tarea.id) {

<li [class.completada]="tarea.completed"

(click)="toggleCompletada(tarea)">

{{ tarea.title }}

</li>

}

</ul>

</div>

Formularios Reactivos: validación desde TypeScript

💡 Con formularios reactivos, defines las reglas de validación en la clase (TypeScript), no en el HTML. Esto hace que sean más fáciles de testear y mantener que los formularios de template.

import { ReactiveFormsModule, FormBuilder } from '@angular/forms';

export class FormularioComponent {

fb = inject(FormBuilder);

form = this.fb.group({

nombre: ['', Validators.required],

email: ['', [Validators.required, Validators.email]],

edad: [0, Validators.min(18)]

});

enviar() {

if (this.form.valid) {

console.log(this.form.value);

}

}

}

Validación en Template

<form [formGroup]="form" (ngSubmit)="enviar()">

<input formControlName="email">

@if (form.get('email')?.invalid &&

form.get('email')?.touched) {

<p class="error">Email inválido</p>

}

<button [disabled]="form.invalid">

Enviar

</button>

</form>

Zoneless Mode — El futuro de Angular

Zone.js era la librería que usaba Angular para detectar cambios automáticamente. Con Signals, ya no la necesitamos: Angular sabe exactamente qué ha cambiado y actualiza sólo eso.

  • Bundles más pequeños (menos código descargado)
  • Arranque más rápido
  • Rendimiento mejorado en listas grandes
  • Compatible con todas las APIs de Signals

Activar Zoneless Mode

// main.ts

import {

provideZonelessChangeDetection

} from '@angular/core';

bootstrapApplication(AppComponent, {

providers: [

provideZonelessChangeDetection()

]

});

Deferrable Views: Lazy Loading en el template

💡 Con @defer puedes decir “no cargues este componente hasta que el usuario llegue a él con el scroll”. Angular lo descarga sólo si y cuando hace falta.

@defer (on viewport) {

<app-grafico-pesado />

} @placeholder {

<p>Cargando gráfico...</p>

} @loading (minimum 1s) {

<app-spinner />

} @error {

<p>Error al cargar</p>

}

Pipes Personalizados

import { Pipe, PipeTransform } from '@angular/core';

@Pipe({

name: 'truncate',

standalone: true

})

export class TruncatePipe implements PipeTransform {

transform(value: string, limit = 20) {

return value.length > limit

? value.substring(0, limit) + '...'

: value;

}

}

❌ Errores Comunes

SOLID en Angular

  • S: Un componente, una responsabilidad
  • O: Extensible sin modificar (herencia)
  • L: Interfaces y abstracciones
  • I: Interfaces específicas, no genéricas
  • D: Inyección de dependencias
